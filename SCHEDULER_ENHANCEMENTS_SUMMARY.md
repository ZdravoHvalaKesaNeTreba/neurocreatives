# Улучшения планировщика: Глубина парсинга и типы запусков

## Дата завершения
18 марта 2026 г.

## Обзор
Реализованы два важных улучшения системы планировщика:
1. **Настройка глубины парсинга** - возможность выбора временного диапазона для парсинга постов
2. **Отслеживание типов запусков** - разделение ручных и автоматических запусков в истории

---

## 1. Настройка глубины парсинга (Parse Depth)

### Описание
Добавлена возможность настройки временного диапазона для парсинга постов из Telegram-каналов.

### Доступные режимы:
- **Только сегодняшние посты** (`today`) - парсятся посты опубликованные сегодня (00:00 - 23:59)
- **За последние 3 дня** (`3days`) - парсятся посты за последние 72 часа
- **С определенной даты** (`from_date`) - парсятся все посты начиная с указанной даты

### Изменения в базе данных:
Добавлены настройки в таблицу `settings`:
- `parse_depth` - режим глубины парсинга (по умолчанию: 'today')
- `parse_from_date` - дата начала парсинга для режима from_date (формат: YYYY-MM-DD)

### Изменения в коде:

#### Backend:
- **`parser/telegram_parser.py`** (Строки 20-78):
  - Обновлена сигнатура метода `parse_channels()`:
    ```python
    async def parse_channels(
        self, channels: list, limit: int = 100,
        parse_depth: str = 'today',
        parse_from_date: str = None
    )
    ```
  - Реализована логика фильтрации по датам для всех трех режимов
  - Удален устаревший параметр `today_only`

- **`api/server.py`** (Строки 292-366):
  - Функция `run_parser_task()` читает настройки parse_depth из БД
  - Передает параметры в парсер при запуске

- **`api/server.py`** (Строки 609-632):
  - GET `/api/schedule` возвращает parse_depth и parse_from_date

- **`api/server.py`** (Строки 682-691):
  - POST `/api/schedule` сохраняет parse_depth и parse_from_date

#### Frontend:
- **`web/templates/index.html`** (Строки 392-413):
  - Добавлен выпадающий список для выбора режима парсинга
  - Добавлен date picker для выбора даты (показывается только для режима from_date)

- **`web/static/app.js`**:
  - Строки 62-70: Загрузка настроек parse_depth в `loadScheduleSettings()`
  - Строки 141-151: Сохранение настроек parse_depth в `saveScheduleSettings()`
  - Строки 281-288: Обработчик изменения режима (показ/скрытие date picker)

## 2. Отслеживание типов запусков (Run Types)

### Описание
Добавлено различение ручных и автоматических запусков парсера в истории.

### Типы запусков:
- **🖱️ Ручной** (`manual`) - запущено пользователем через UI
- **⏰ Авто** (`auto`) - запущено автоматически по расписанию

### Изменения в базе данных:
- **`db/models.py`** (Строка 76):
  - Добавлено поле `run_type` в модель `ScheduleLog`:
    ```python
    run_type = Column(String(20), default='auto', nullable=False)
    ```

- **Миграция**: Создан скрипт `db/migrate_add_run_type_sql.py`
  - Универсальная поддержка PostgreSQL и SQLite
  - Использует `inspect()` для проверки существующих колонок
  - SQL: `ALTER TABLE schedule_logs ADD COLUMN run_type VARCHAR(20) DEFAULT 'auto' NOT NULL`

### Изменения в коде:

#### Backend:
- **`api/server.py`** (Строка 292):
  - Добавлен параметр `run_type: str = 'manual'` в `run_parser_task()`
  - Логирование выполнения с указанием типа запуска

- **`api/server.py`** (Строки 746-755):
  - GET `/api/schedule/logs` возвращает run_type для каждой записи
  - Обратная совместимость: `run_type='auto'` по умолчанию для старых записей

- **`scheduler/scheduler.py`** (Строки 34-52):
  - Функция `_run_scheduled_task()` вызывает `run_parser_task(run_type='auto')`
  - Упрощена логика: вместо дублирования кода используется единая точка входа

#### Frontend:
- **`web/templates/index.html`**:
  - Строка 247: Заголовок изменен с "История автозапусков" на "История запусков"
  - Строка 257: Добавлена колонка "Тип" в таблицу истории

- **`web/static/app.js`**:
  - Строки 196-207: Отображение типа запуска в `loadScheduleLogs()`
  - Строка 228: Исправлен colspan с 5 на 6 (добавлена колонка)

## Инструкция по тестированию

### 1. Тестирование глубины парсинга:

```bash
# Запустить приложение
./venv/bin/python run.py
```

1. Открыть настройки (иконка ⚙️)
2. Перейти на вкладку "Планировщик"
3. Выбрать режим "За последние 3 дня"
4. Сохранить и запустить парсер вручную
5. Проверить в логах, что парсятся посты за 3 дня

6. Выбрать режим "С определенной даты"
7. Указать дату (например, 01.03.2026)
8. Сохранить и запустить парсер
9. Проверить, что парсятся все посты начиная с указанной даты

### 2. Тестирование типов запусков:

1. Запустить парсер вручную (кнопка "Запустить парсер")
2. Перейти на вкладку "Логи"
3. Проверить, что последняя запись имеет тип "🖱️ Ручной"

4. Настроить планировщик с частотой "каждый час"
5. Включить планировщик
6. Дождаться автоматического запуска
7. Проверить, что запись имеет тип "⏰ Авто"

## Обратная совместимость

✅ Все изменения обратно совместимы:
- Существующие записи в БД получают run_type='auto' по умолчанию
- API возвращает run_type даже для старых записей (hasattr проверка)
- Если parse_depth не указан, используется 'today' (текущее поведение)
- Миграция безопасна: проверяет наличие колонки перед добавлением

## Известные ограничения

1. **Timezone**: Все даты обрабатываются в UTC, но отображаются в часовом поясе пользователя
2. **Производительность**: При режиме from_date с очень старой датой парсинг может занять больше времени
3. **Валидация**: Дата from_date не валидируется на стороне клиента (только на сервере)
