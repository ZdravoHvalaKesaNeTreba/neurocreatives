# Тестирование планировщика автоматического парсинга и анализа

## Как проверить, что планировщик работает корректно

### 1. Быстрое тестирование с режимом "каждые 5 минут"

Для удобного тестирования добавлен режим **"Каждые 5 минут"**, который запускает задачи каждые 5 минут.

**Шаги для теста:**

1. Откройте **Настройки** → вкладка **"Планировщик"**

2. Включите планировщик (поставьте галочку "Включить планировщик")

3. Выберите периодичность: **"Каждые 5 минут (для тестирования)"**

4. Оставьте срок действия: **"Бессрочно"**

5. Нажмите **"Сохранить"**

6. Проверьте статус:
   - Должен появиться статус "Запущен" (зеленый индикатор)
   - Должно отобразиться время следующего запуска (примерно через 5 минут)

### 2. Мониторинг выполнения

#### Через UI (вкладка "Логи"):

1. Перейдите на вкладку **"Логи"**

2. В верхней части будет виджет **"История автозапусков планировщика"**

3. После каждого автозапуска здесь появится запись с:
   - Датой и временем выполнения
   - Статусом (Успешно/Ошибка)
   - Количеством спарсенных изображений
   - Количеством проанализированных изображений
   - Деталями (если были ошибки)

4. В системных логах ниже вы увидите сообщения типа:
   ```
   🕐 Запуск запланированной задачи парсинга и анализа...
   📡 Запуск парсера...
   ✅ Парсинг завершен. Собрано постов: X
   🔍 Запуск AI анализа...
   ✅ Анализ завершен. Проанализировано: X
   📝 Лог выполнения сохранен (статус: success)
   ```

#### Через терминал (логи бэкенда):

Следите за терминалом, где запущен сервер. При каждом запуске вы увидите:

```
INFO:scheduler.scheduler:🕐 Запуск запланированной задачи парсинга и анализа...
INFO:scheduler.scheduler:📡 Запуск парсера...
INFO:scheduler.scheduler:✅ Парсинг завершен. Собрано постов: X
INFO:scheduler.scheduler:🔍 Запуск AI анализа...
INFO:scheduler.scheduler:✅ Анализ завершен. Проанализировано: X
INFO:scheduler.scheduler:📝 Лог выполнения сохранен (статус: success)
```

### 3. Проверка базы данных

Если хотите проверить напрямую в БД:

```sql
-- Посмотреть настройки планировщика
SELECT * FROM settings WHERE key LIKE 'schedule_%';

-- Посмотреть логи выполнения
SELECT * FROM schedule_log ORDER BY timestamp DESC LIMIT 10;
```

### 4. Проверка разных режимов периодичности

После успешного теста с "каждый час", попробуйте другие режимы:

#### Ежедневно
- Запуск каждый день в указанное время (например, 08:00 МСК)

#### Еженедельно
- Запуск каждый понедельник в указанное время

#### Раз в 2 недели
- Запуск каждые 2 недели по понедельникам

#### Ежемесячно
- Запуск 1 числа каждого месяца в указанное время

#### Выбрать дни недели (Custom)
- Выберите конкретные дни недели (например, Пн, Ср, Пт)
- Запуск будет происходить в эти дни в указанное время

### 5. Проверка срока действия

Протестируйте срок действия:

1. **Бессрочно** - планировщик работает без ограничений

2. **До даты** - укажите дату окончания:
   - Планировщик автоматически прекратит работу после этой даты
   - Проверьте, что после указанной даты задачи не выполняются

### 6. Проверка отключения планировщика

1. Снимите галочку "Включить планировщик"
2. Нажмите "Сохранить"
3. Проверьте, что:
   - Статус изменился на "Не запущен"
   - Время следующего запуска исчезло
   - Новые автозапуски не происходят

## API эндпоинты для тестирования

### Получить текущие настройки расписания
```bash
curl http://localhost:8000/api/schedule
```

### Обновить расписание
```bash
curl -X POST http://localhost:8000/api/schedule \
  -H "Content-Type: application/json" \
  -d '{
    "enabled": true,
    "frequency": "every_5_minutes",
    "time": "08:00",
    "days": [],
    "end_type": "indefinite",
    "end_date": null
  }'
```

### Получить логи автозапусков
```bash
curl http://localhost:8000/api/schedule/logs?limit=10
```

## Возможные проблемы и решения

### Планировщик не запускается

**Проблема:** Статус показывает "Не запущен" даже после включения

**Решение:**
1. Проверьте логи бэкенда на наличие ошибок
2. Перезапустите сервер
3. Убедитесь, что APScheduler установлен (`pip install apscheduler`)

### Задачи не выполняются по расписанию

**Проблема:** Время следующего запуска отображается, но задачи не запускаются

**Решение:**
1. Проверьте таймзону - все настроено для Europe/Moscow (UTC+3)
2. Убедитесь, что сервер не перезагружается между запусками
3. Проверьте логи на наличие ошибок при выполнении задач

### Ошибки при парсинге или анализе

**Проблема:** В логах автозапусков статус "Ошибка"

**Решение:**
1. Проверьте детали ошибки в столбце "Детали"
2. Убедитесь, что:
   - Telegram API настроен корректно (API_ID, API_HASH)
   - OpenAI API ключ указан в настройках
   - Есть доступ к интернету
3. Проверьте системные логи для подробностей

## Рекомендации по использованию

1. **Для production:** Используйте режимы "Ежедневно" или "Еженедельно"

2. **Для тестирования:** Используйте режим "Каждые 5 минут"

3. **Время запуска:** Выбирайте время низкой нагрузки (например, раннее утро)

4. **Мониторинг:** Регулярно проверяйте историю автозапусков на наличие ошибок

5. **Срок действия:** Используйте "До даты" для временных кампаний или тестирования

## Примечания

- Все настройки времени используют таймзону **Europe/Moscow (МСК, UTC+3)**
- Планировщик автоматически запускается при старте приложения
- При изменении настроек расписание обновляется немедленно
- Планировщик корректно останавливается при завершении приложения
- Максимум одна задача может выполняться одновременно (max_instances=1)
